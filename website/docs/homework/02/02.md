# Homework 2: Rendering & Persistence

This homework is a guided build. You will start from starter code, make focused changes in specific files, and then show evidence of what you built. The goal is to practice server rendering with HTMX, a repository boundary, and Prisma persistence in one small project.

The homework opens on **March 5, 2026** and is due on **March 24, 2026 at 11:59 PM**. This range includes spring break, so plan your time early. The expected work time is about 10 to 12 hours.

## Start Up Kit

Start by downloading `hw-02.zip` from Canvas. Unzip it so you have a folder named `hw-02`, then do all work inside that folder. Do not build from scratch.

## Setup

Open a terminal, move into `hw-02`, install dependencies, and prepare Prisma. Use these exact commands:

```bash
cd hw-02
npm install
cp .env.example .env
npm run prisma:generate
npm run prisma:migrate
npm run dev
```

At this point the starter app should run. Before you change anything, click around and understand what is already wired.

## What You Must Build

Your job is to turn the starter into a complete homework solution by finishing the `TODO(HW2)` work and extending the app behavior in a clear way. You must keep the repository boundary design. Routes should handle request and response work, while repository classes handle data storage work.

Use this rule while building: when you change a feature, also update one test or add one test that proves the behavior.

## Step 1: Strengthen the Data Model

Open `prisma/schema.prisma` and improve the `Entry` model. Add one meaningful field that helps your feature and supports an invariant. For example, you might add a short status field or a label field. Keep the model simple, but make sure it reflects your design choices.

After editing the schema, run migration again so the database matches your model.

## Step 2: Enforce Invariants in Memory Repository

Open `src/repository/InMemoryEntryRepository.ts` and complete the `TODO(HW2)` logic. Add clear checks before data is saved. Your checks should enforce at least two invariants. A good example is requiring non-empty title and body with a minimum length. If data is invalid, return a clear error path (throw or structured result) and handle it consistently.

This is important because the in-memory repository is your first implementation of persistence rules.

## Step 3: Enforce Invariants in Prisma Repository

Open `src/repository/PrismaEntryRepository.ts` and complete the `TODO(HW2)` logic there too. The same core invariants should hold in both repository implementations. If you handle known Prisma errors, map them to clear app-level behavior so route code can respond predictably.

This step ensures your rules do not depend on one storage mode only.

## Step 4: Improve Route Behavior and HTMX Responses

Open `src/app.ts` and complete the route TODO work. Make your create and toggle flows return clean partial HTML updates for HTMX. Error messages should be easy to understand and should render in a consistent place. Keep the trigger-request-fragment-swap pattern clear in your code and template usage.

Your final app must include at least three HTMX interactions that work without full page reloads. The search interaction in the starter can count as one if you keep it working correctly.

## Step 5: Confirm Storage Boundary Switching

Open `src/server.ts` and verify both modes work. The app should run with `REPO_MODE=memory` and with `REPO_MODE=prisma` without route rewrites. This is part of the architecture goal for this homework.

Run both modes locally and confirm behavior is consistent.

## Step 6: Write Real Tests

Open `tests/hw2-contract.test.ts` and replace the placeholder with real tests. Your tests must show create/list behavior, at least one invariant check, and restart-oriented persistence reasoning. If you cannot fully automate restart in your current setup, include a clear test note in your write-up and still add meaningful automated checks.

Your test file should directly support claims you make in the PDF.

## Step 7: Generate Evidence Report

When your implementation is ready, run the assignment tool from the `hw-02` folder:

```bash
npm run hw2:evidence
```

This command runs `hw-02/.vscode/evidence.mjs` and writes output into `.reports/`. Do not manually edit `.reports/hw2-evidence.md`.

## Step 8: Write the PDF Report

Now write `hw2-writeup.pdf`. This is a required technical part of the assignment, not an optional reflection.

Your PDF should read like a short technical explanation. In plain words, explain your canonical state, your invariants, and why your repository boundary is in the right place. Then explain one HTMX interaction from trigger to swap, explain how your design handles restart correctness, and explain one failure or race risk and how you handled it. Include one SQL-to-Prisma mapping from your own code.

Your PDF must also include one diagram and at least three screenshots with captions. One screenshot should show HTMX partial update behavior. One should show persistence evidence such as `sqlite3` output or app-visible database state. One should show your test run.

If you used AI tools, include a brief and honest disclosure in the PDF describing where you used them and what you personally verified.

## What You Submit to Canvas

You will submit exactly two files. The first file is `hw2-evidence.md`, which is taken from `.reports/hw2-evidence.md`. The second file is `hw2-writeup.pdf`.

## Required Evidence in `hw2-writeup.pdf`

Your writeup must include all items in this list.

1. One architecture or data model diagram that names your main components and persistence boundary.
2. One diagram or visual flow that explains one HTMX interaction from trigger to swap.
3. Screenshot of the running app showing a successful HTMX partial update.
4. Screenshot of an error case in the app with the error message visible.
5. Screenshot of persistence evidence using `sqlite3` output or app-visible database rows.
6. Screenshot of test execution output showing your homework tests ran.
7. Screenshot or note showing both repository modes were exercised (`REPO_MODE=memory` and `REPO_MODE=prisma`).
8. Short caption under each screenshot explaining what the grader should look at.
