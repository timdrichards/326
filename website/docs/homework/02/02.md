## Feature Plan: Write Your Invariants First

After you choose your feature, but before you start the steps below, write the invariants for that feature. Invariants are rules that must stay true every time your app runs.

Write down more than the minimum if you can. The more clearly you define these rules up front, the easier it will be to implement the feature code and to write focused tests later. Invariants give you a direct checklist for both coding and testing.

You are required to include these invariants in your final submission. In your PDF, include a `Feature Invariants` section and write each invariant using exactly the notation shown below.

Use this notation for each invariant:

`INV-<number>: IF <condition>, THEN <required rule>, BECAUSE <why this rule matters>.`

You can also add a short pass/fail example under each one:

`Pass example: <valid case>`

`Fail example: <invalid case>`

Example (from the walkthrough feature only):

```
INV-1: IF a status filter is applied,
  THEN only entries with that status are shown,
  BECAUSE users should never see mixed results when filtering.
Pass example: filter=completed shows only completed entries.
Fail example: filter=completed still shows active entries.

INV-2: IF a filter value is not one of all/active/completed,
  THEN the system returns a typed validation error,
  BECAUSE unknown filter values should not silently produce incorrect output.
Pass example: filter=active returns the filtered list.
Fail example: filter=done returns success with unfiltered data.

INV-3: IF the same filter request is run in
  REPO_MODE=memory and REPO_MODE=prisma,
    THEN the visible result set is the same,
    BECAUSE behavior must be storage-agnostic.

Pass example: both modes show the same IDs for filter=active.
Fail example: memory mode and prisma mode show different results
              for the same filter.
```

## Step 2: Strengthen the Data Model

Open `prisma/schema.prisma` and improve the `Entry` model with one additional model-level choice tied to your invariants. A model-level choice means a schema decision on the model itself, such as adding a field, adding a default value, making a field unique, or changing nullability. Keep the model simple, but make sure this choice reflects your design reasoning.

Implement this step by doing three things. First, decide what new rule your data should guarantee over time, then encode that rule in the schema. For example, if you want every entry to have a stable category, add a field with a sensible default. Second, make sure your schema decision is something your service and repository code can enforce consistently in both memory and Prisma modes. Third, confirm the new model shape is reflected in your create/read flows so your app behavior and database design stay aligned.

Example for the walkthrough feature (status filter): if your feature needs clear status states, you might make status explicit in the model, such as adding a `status` field with a default value (`"active"`). In Prisma terms, this is a model-level choice because the schema now guarantees each entry has a known status value at creation time. After that schema change, your Step 3 and Step 4 logic would enforce valid status transitions and reject unknown values.

Example snippet in `schema.prisma`:

```prisma
model Entry {
  id          Int      @id @default(autoincrement())
  title       String
  body        String
  status      String   @default("active")
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}
```

After editing the schema, run this command so the database matches your model:

```bash
npm run prisma:migrate
```

This command creates and applies a migration based on your schema changes. It updates the actual SQLite database structure (tables/columns/defaults) so your running app and your Prisma schema stay in sync. You need this because changing `schema.prisma` alone does not change the database file.

After migration, regenerate the Prisma client and verify the app still runs with your updated model:

```bash
npm run prisma:generate
```

This command rebuilds the generated Prisma client code from your current schema. You need this so TypeScript types and Prisma query APIs reflect your new fields and model changes; otherwise your app may compile or run with stale generated code.

Helpful Prisma docs for this step:

- [Prisma schema models](https://docs.prisma.io/docs/orm/prisma-schema/data-model/models)
- [Prisma Migrate overview](https://docs.prisma.io/docs/orm/prisma-migrate)
- [Prisma migrate CLI reference](https://docs.prisma.io/docs/cli/migrate)

## Step 3: Enforce Invariants in Service and Memory Repository

Open `src/service/EntryService.ts` and `src/repository/InMemoryEntryRepository.ts` and implement clear checks before data is saved. Implement as many of your planned invariants as you can, with a required minimum of 3 invariants. For the walkthrough-style feature, a good example would be validating that status values are only `all`, `active`, or `completed`, and returning a typed error for anything else.

We start with service and memory mode first on purpose. This gives you a fast feedback loop for core domain rules before database details are involved. If your invariants are not clear in memory mode, they will be harder to debug once Prisma and migrations are added to the flow.

You are also required to update the memory repository, not just Prisma, because the assignment expects both repository modes to behave the same way. The memory repository is part of the contract boundary and test story. Keeping its guards and return behavior aligned with service rules proves your architecture is storage-agnostic instead of database-dependent.

Implement this step by doing three things. First, write service-level validation that normalizes input and returns typed `Err(...)` values when invariants fail. Second, keep repository-level guard checks in the in-memory repository so invalid data is still rejected even if service logic is bypassed. Third, make sure success paths return `Ok(...)` values with consistent shape so controller behavior stays predictable.

At least one of your Step 3 checks must directly relate to the model-level choice you made in Step 2. In other words, if you changed the schema, your validation rules should enforce that same design decision in runtime behavior.

Example: if you added a `status` field with a default like `"active"` in Step 2, then in Step 3 you should validate a clear rule such as "status must be one of `active` or `completed` for stored entries, and filter input must be one of `all`, `active`, or `completed`." If the value is invalid, return `Err(ValidationError(...))`; if it is missing where your design allows a default, apply your default behavior consistently.

Do not throw generic errors from this flow. Return typed `Err(...)` values so controller logic can map outcomes consistently.

## Step 4: Enforce Invariants in Prisma Repository

Open `src/repository/PrismaEntryRepository.ts` and make the Prisma-backed behavior match the same rules you enforced for the in-memory path. This means create, read, search, and toggle behavior should feel the same to the controller no matter which storage mode is active.

Implement this step by doing three things. First, normalize and validate incoming values before writing to the database so your persisted records follow your invariants. Second, map Prisma reads and writes into your app `Entry` shape consistently, including date handling and sort behavior. Third, handle expected failure cases by returning typed `Err(...)` values, including not-found cases and database dependency failures, so the controller can map them to predictable HTTP responses.

In this step, you should also write tests that run against the Prisma repository. Most of these tests will mirror your memory-repository tests, because the expected behavior should be the same.

A straight forward approach is to write direct tests, one for memory and one for Prisma, with clear setup and assertions:

```ts
it('memory repo rejects invalid status filter', async () => {
  const service = CreateEntryService(CreateInMemoryEntryRepository())
  const result = await service.filterEntriesByStatus('done')
  expect(result.ok).toBe(false)
})

describe('prisma repo status filter tests', () => {
  const prisma = new PrismaClient(/* adapter config */)
  const service = CreateEntryService(CreatePrismaEntryRepository(prisma))

  beforeAll(async () => {
    await prisma.$connect()
  })

  beforeEach(async () => {
    // Reset and pre-populate known data before each test.
    await prisma.entry.deleteMany()
    await prisma.entry.createMany({
      data: [
        { title: 'A', body: 'Entry A body', status: 'active' },
        { title: 'B', body: 'Entry B body', status: 'completed' },
        { title: 'C', body: 'Entry C body', status: 'active' },
      ],
    })
  })

  afterAll(async () => {
    await prisma.$disconnect()
  })

  it('rejects invalid status filter', async () => {
    const result = await service.filterEntriesByStatus('done')
    expect(result.ok).toBe(false)
  })

  it('returns completed entries and confirms DB contains seeded data', async () => {
    const seededCount = await prisma.entry.count()
    expect(seededCount).toBe(3)

    const result = await service.filterEntriesByStatus('completed')
    expect(result.ok).toBe(true)
    if (result.ok) {
      expect(result.value.length).toBe(1)
      expect(result.value[0].status).toBe('completed')
    }
  })
})
```

What this setup does and why: `beforeEach` resets and seeds the database so each test starts from a known state. That keeps tests repeatable and easier to debug. The `count()` check proves the database actually has data, and the filter test proves your Prisma repository is reading stored data and applying the feature invariants.

This step ensures your rules do not depend on one storage mode only, and your tests should prove that.

After you finish this step, run your unit tests again and confirm your Prisma-backed flow passes the same kinds of checks as memory mode, including at least one success path and one failing-input or failure-path case.
